# Reconcile pre-existing plantimage.uuid migration-state drift

## Problem

`makemigrations garden_calendar` always wants to emit an `AlterField` on
`PlantImage.uuid` even with no model change — the migration-recorded field state
diverges from the model definition. This is a phantom/no-op migration: harmless
at runtime, but it surprises anyone who runs `makemigrations` and makes it look
like they forgot to commit a migration.

## Findings

- Discovered while implementing **todo 205** (M1): `makemigrations
  garden_calendar` for the CareLog change also bundled an unrelated
  `~ Alter field uuid on plantimage`. That drift was deliberately stripped from
  205's migration (kept surgical); it remains unreconciled.
- Confirmed standalone:

  ```text
  $ python manage.py makemigrations garden_calendar --dry-run
    apps/garden_calendar/migrations/0007_alter_plantimage_uuid.py
      ~ Alter field uuid on plantimage
  ```

- **Root cause.** The model (`apps/garden_calendar/models.py:964-970`) declares:

  ```python
  uuid = models.UUIDField(
      default=uuid.uuid4, editable=False, unique=True,
      primary_key=True, help_text="Unique identifier for secure references",
  )
  ```

  Migration `0005_add_plantimage_uuid_primary_key` records the field state via an
  `AlterField` (step 3, `:128-137`) that has `unique=True` but **not**
  `primary_key=True`, then swaps the real DB primary key through a
  `SeparateDatabaseAndState` block (`:140+`). The recorded migration state never
  ends up exactly matching the model's `primary_key=True` (+ implicit
  `serialize=False`), so the autodetector keeps proposing a reconciling
  `AlterField`.
- **Benign:** the DB already has `uuid` as the real primary key (the 0005 SQL
  swap did that). This is a Django *state* mismatch only — no column change, no
  data risk.

## Recommended Action

1. Generate the reconciling migration: `python manage.py makemigrations
   garden_calendar` → produces `0007_alter_plantimage_uuid.py` (state-only
   `AlterField`). Commit it as-is.
2. Verify `makemigrations garden_calendar --check --dry-run` then reports
   **"No changes detected"**.
3. Optional cleanup: drop the redundant `unique=True` from the model field
   (`primary_key=True` already implies unique) only if it doesn't itself
   re-trigger drift — verify with the `--check` in step 2.
