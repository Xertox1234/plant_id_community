## The question this audit answers

Not "does each hook run without error" (plumbing — yes) but **"does the harness
actually produce higher-quality code?"** Lens: the *recurrence test* (did codified
rules prevent repeat bugs?), the *gate-bypass* surface, and *coverage gaps* where
bad code reaches `main` ungated.

## Headline verdict

**The agent-facing harness is genuinely well-built and working. The real
quality risk is CI test coverage — functional tests are gated for mobile only.**

What works (verified):

- JIT mistake-injection fires **end-to-end** — a `.tsx` edit importing bare
  `react-router` produced the correct write-time warning; the clean variant was
  silent; 34 matcher tests pass.
- A **real recurrence lesson is encoded as a working write-time guard**: 5 forum
  `@action` endpoints shipped without rate limits (todos 104–109), and that exact
  class is now the content-matched trigger `drf-action-no-ratelimit`
  (`source: todos/archive 104-109`), which fires correctly in the matcher.
  *Caveats (load-bearing for F3):* this trigger was **hand-seeded in the v1-spine
  commit** (46234f1, severity `warn`), not produced by the *automated* capture
  path — all 7 triggers are `warn`; zero auto-captured `candidate`s exist yet —
  and there is **no fire telemetry**, so it is unproven that any trigger has fired
  in a real session. The mechanism is sound and tested; its autonomous,
  in-the-wild prevention is not yet demonstrated.
- Rule-injection coverage is sound (forum_integration / garden / users →
  `api,security,database`; `.tsx` → `react,typescript`; firebase →
  `firebase,security`); all 6 trigger `pattern_ref`s resolve; all 19 CLAUDE.md
  pattern-doc refs resolve; `rate-limiting.md` exists (32 KB).
- Agent fleet is consistent (10 reviewers share Scope/severity/JSON; sound
  haiku-orchestrator / sonnet-reviewer split); the two orchestrators are
  **complementary by design** (incremental git-diff vs full-repo), not redundant.

Where bad code reaches `main` ungated:

- **Backend tests** (F2) and **web type-check + tests** (F1) run in **no** CI
  workflow. Web has no CI workflow at all (only `npm audit`). Backend/web lint
  lives only in bypassable pre-commit.
- The **harness's own** (passing) tests aren't gated either (F3).
- The one blocking gate (kimi-review) is pre-authorized to be skipped and is
  fail-open locally (F4).

## Meta-finding — oversized batches produced unreliable, misattributed results

Several early tool results were **internally inconsistent with later isolated
re-runs**: a phantom `print("placeholder")` / 95-line count for a 201-line file; a
`def main(): return _run()` body `grep` later proved absent; a `(NONE)`
rule-coverage result for a file that gets `firebase,security`; a "230 tests, 3
FAILED" result (with detailed-but-nonexistent failure bodies — the extraction file
came back empty) for a 34-test green suite; a grep claiming backend CI runs
tests+flake8+black when it runs none; and garbled duplicate lines in two later
greps. **Four would have become published findings (two HIGH); one caused an
over-correction that nearly buried a real finding (F2).**

**Likely mechanism (honest about uncertainty):** the affected results all came
from **large single-message batches** (20+ tool calls, including duplicate reads,
dependent/sequential calls, two malformed `AskUserQuestion` calls, and filler
"noop" calls). Every claim re-checked in a **small, isolated, file-backed**
invocation was consistent and correct. I cannot prove the relay "fabricates"
content; the evidence better fits **cross-wiring / misattribution across an
oversized parallel batch** than infrastructure fabrication.

**Actionable lessons (for this harness and future audits):**

1. Keep verification batches **small and independent**; never mix dependent or
   duplicate calls into one block — they can cross-wire.
2. Prefer **direct primary-source reads** (`Read` of the file) over grep/subagent
   summaries for any load-bearing claim; corroborate quantitative claims ≥2 ways.
3. Route important results through **files** (`> /tmp/x; Read x`) rather than
   inline stdout, which proved more stable.
A "correction" built on a single unreliable read can be as wrong as the original
(it nearly buried F2). Verification must be plural and prefer primary sources.

## Findings Detail

### F1 — No web CI; web TypeScript + tests fully ungated (HIGH)

`.github/workflows/` = `backend-ci.yml`, `mobile-ci.yml`, `security-scan.yml`,
`kimi-review.yml` — no `web-ci.yml`. The web stack in CI gets only `npm audit`
(`security-scan.yml:70` frontend-security). No `tsc`, no vitest, no playwright in
any workflow (negative grep guard fired). Pre-commit web hooks are **eslint +
prettier only** (`.pre-commit-config.yaml:151-165`) — no type-check, no tests —
and pre-commit is bypassable with `--no-verify`. Yet `web/package.json` defines
`type-check` (`tsc --noEmit`), `test` (vitest), and `test:e2e` (playwright), and
`web/vitest.config.ts` + `web/playwright.config.ts` exist. A TypeScript type error
or a failing web test merges to `main` with zero gate.

### F2 — Backend test suite runs in no CI workflow (HIGH)

`backend-ci.yml` (direct read, lines 1–72) runs `pip check`,
`python manage.py check`, and `python manage.py spectacular` — **not**
`manage.py test`. Pre-commit runs no backend tests either (black/flake8/isort
only). Mobile (`mobile-ci.yml:46-47` `flutter test`) is the only stack whose
tests run in CI. A PR that breaks backend tests merges green. Likely partly
deliberate (the suite needs postgres+redis; `backend-ci.yml` uses sqlite for the
light checks) — so the fix is a real test job with services, not a hand-wave.

### F3 — Harness tests pass but nothing gates them (MEDIUM)

Hook tests (46 assertions) + matcher/capture python tests (34+) are green, but
`grep -rl scripts/inject .github/workflows` and `grep -rl hooks/test
.github/workflows` are both empty. A future edit that breaks a hook's `jq` JSON
emission would silently disable rule injection for every session, uncaught.

### F4 — The one blocking gate is pre-authorized to be skipped (MEDIUM)

`kimi-review` can block a commit (deny on engine exit 2), but
`.claude/settings.local.json:71` allow-lists `Bash(SKIP_KIMI_REVIEW=1 git commit
*)` (no prompt), and `scripts/kimi-precommit.sh` is fail-open
(`:15` skips on the env var; `:78` exits 0 on any non-CRITICAL status). The
CI `kimi-review.yml` fails *closed* but skips entirely if `OPENROUTER_API_KEY` is
unset (`:40-43`). The env-var skip leaves no commit trace, so the true skip rate
is unmeasurable — a structural observation. Memory `commit_hook_friction`
documents friction (whole-file reformat, todo 117 open) that pushes toward
skipping. **Do not weaken the CRITICAL block — reduce friction instead.**

### F5 — CLAUDE.md forum path is wrong (LOW)

`CLAUDE.md:18` lists `backend/apps/forum/`; the real, *tracked* app is
`backend/apps/forum_integration/` (`apps/forum/` is untracked — a decoy, per
memory `forum_app_path`). Misleads any agent navigating from the canonical doc.

### F6 — Stale handoff artifacts (LOW)

`inject-patterns.sh:8-19` still reads "PROPOSED v1 … ships as a handoff artifact.
Apply: `cp scripts/inject/inject-patterns.sh.proposed …`" — but `diff -q` shows
the live hook is IDENTICAL to the `.proposed` file (already applied). Same for
`codify-SKILL.md.proposed`; `codify-capture-step.md` is a dead application guide.
A reader cannot tell applied from pending.

### F7 — Stale locked worktrees (LOW)

`git worktree list` shows two `locked` agent worktrees
(`.claude/worktrees/agent-ad69486e286e62700` — a full repo checkout — and
`agent-aff2d3d2efe25dff5`). The first pollutes repo-wide `find`/tree output.
Inspect for unmerged work, then prune.

### F8 — Prevention is unmeasurable (MEDIUM)

This is the most direct answer to the audit's brief ("how effective is the
harness?"): **currently unanswerable.** `match_triggers.py` writes matched
warnings to stdout only — no persistent fire log; `inject-patterns.sh` records
nothing about which triggers fired. All 7 triggers are hand-seeded `severity:
warn` (Counter = `{warn: 7}`) from the v1-spine commit 46234f1; **zero**
auto-captured `candidate` triggers exist, so the automated codify→capture→trigger
path has never completed end-to-end in the wild. There is no evidence any trigger
has fired in a real session or prevented a real recurrence. The mechanism is
sound and unit-tested; its in-the-wild effectiveness is simply unknown — and an
unmeasured guardrail can rot silently. Fix: lightweight fire telemetry + a
counts reporter (todo 133).
