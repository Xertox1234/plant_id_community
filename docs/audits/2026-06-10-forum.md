# Audit: Forum Extension (wagtail_forum + forum_host)

> **Date:** 2026-06-10
> **Trigger:** User-invoked `/audit` on the forum extension — "make sure we did everything right." Fills the deliberate coverage gap noted in the 2026-06-09 maintainability audit (the new forum package was skipped because the churn signal pointed at the deleted `forum_integration/`).
> **Domains:** security, django-drf, wagtail, performance, testing
> **Scope:** `backend/packages/wagtail_forum/`, `backend/apps/forum_host/` (models, permissions, API, spam/moderation, tests). Celery reviewer skipped — no Celery usage in scope (`SpamCheckTask` is a Wagtail workflow task).
> **Baseline:** full backend suite green (exit 0) | `manage.py check`: 0 issues
> **Discovery agents:** security, django-drf, wagtail, performance, api-design, test-quality (6 parallel). Heavy independent convergence: rate limiting (4 agents), topic_count undercount (2), unpublish reconciliation (2), title spam bypass (2), bootstrap clobber (2), reply idempotency (3), image-PK reference (2).
> **Research (Phase 2.5):** 2 docs-researchers (wagtail, drf/api) — 13 doc-dependent claims checked, all `confirmed` or `better-fix`, zero contradicted.

## Findings

Each finding has a lifecycle: `open` → `fixing` → `verified` or `deferred` or `false-positive`.

**Status key:**

- `open` — Found but not yet addressed
- `fixing` — Work in progress
- `verified` — Fix applied AND confirmed by test/grep/type-check
- `deferred` — Intentionally postponed (must link to todo)
- `false-positive` — Agent was wrong or issue was already fixed

**Research key** (Phase 2.5 verdict, recorded in the `Research` column):

- `confirmed` — current documentation agrees the finding is valid
- `better-fix` — finding is real, but current docs show a cleaner fix (described in the `Verification` column for Phase 3 to use)
- `contradicted ⚠` — current docs say the flagged pattern is fine; may be a false positive — decide at triage
- `—` — research not applicable, or finding predates Phase 2.5

publish/unpublish must refresh the author's `post_count` — but `post_count` counts live

`transaction.atomic()`. Cascade warning FP — `topic_id` is a local attr; lookups are
`.filter().first()` / no-op `.update()`.

## Phase 6: Orchestrated Code Review (Round 2)

The code-review-orchestrator (6 domain reviewers over the full audit diff) found
**0 critical, 3 high, 9 medium, 9 low** in the Round 1 fix set. All HIGHs and the
actionable MEDIUMs were fixed in Round 2 (kimi-review clean):

- **R2-1 (HIGH, fixed):** `@receiver(post_delete)` had no `sender` — matched every
  model project-wide and disabled Django's fast-delete everywhere. Now two sendered
  receivers (`wagtail_forum.Topic`/`.Post`) + a thread-local pre_delete marker that
  dedups the cascade (a 200-reply topic no longer recounts the board 200×; also
  closes review finding 9).
- **R2-2 (HIGH, fixed):** reconciliation could write `last_post_at = NULL` on a live
  topic (only live post removed) — breaking the cursor-pagination non-null invariant
  (NULLS-FIRST float + `str(None)` cursor 500). Now `Coalesce(..., created_at)` +
  `first_published_at__isnull=False` filter; regression test added.
- **R2-3 (HIGH, fixed):** non-string rich-text block value reached `nh3.clean()` →
  execution-proven TypeError 500 on both write endpoints. `validate_forum_body` now
  type-checks every block value (str for text blocks, dict-of-str for struct);
  covers review finding 4 too. 5-case test added.
- **R2-5 (MED, fixed):** slug auto-suffix could exceed `SlugField(max_length=255)` →
  Postgres DataError 500 (masked by SQLite). Base slug truncated to fit; test added.
- **R2-6 (MED, fixed):** counts/trust ignored TOPIC liveness — a topic-level takedown
  left `board.post_count` and the spammer's autopublish trust funded by hidden posts.
  `topic__live=True` added to both definitions + `_refresh_topic_authors()` on topic
  publish/unpublish/delete; takedown + delete reconciliation tests added.
- **R2-7 (MED, fixed):** idempotency fingerprint excluded route params (same key+body
  on a different board/topic/post silently replayed the wrong resource). Route params
  now mixed into the fingerprint; computed only when a key is present (review 18).
- **R2-8 (MED, fixed):** no in-flight reservation — concurrent same-key retries could
  create a suffixed duplicate. `reserve()` (`cache.add`, 60s sentinel TTL, placed
  after validation so a 400 never wedges the key) → 409 per the IETF draft; test added.
- **R2-10/11/12 (MED, fixed):** test gaps closed — search board-takedown negative,
  sync truncation (`has_more`/boundary `next_since` with monkeypatched page size),
  per-user-not-global throttle isolation, parity test now pins wrapped view callbacks
  (review 21).
- **R2-17 (LOW, fixed):** sync partial index regenerated as `(updated_at, id)` to
  match the query's sort (migration 0008 was uncommitted — replaced in place).
- **Deferred to todo 231 (API-contract LOWs):** sync `board` param silent-empty on
  typo/ambiguity (13), ancestor-liveness visibility (14), reaction cascade-delete
  recount on user deletion (16), 429/422/400 OpenAPI response-code gaps (19/20),
  idempotency endpoint-scope isolation test (21 tail).
- **Acknowledged, documented in signals.py (15):** host notifications fire inside the
  publish transaction; receivers doing async work must use `transaction.on_commit`
  (changing it now would alter the synchronous test contract for no current consumer).
